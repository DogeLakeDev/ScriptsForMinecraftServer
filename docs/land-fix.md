# 当前修复清单

## 目标

统一修复领地系统和经济系统中的数据一致性、权限安全、缓存刷新、事务原子性和用户体验问题。

优先级：

- P0：可能造成经济损失、权限越权或数据破坏
- P1：会导致功能失效、状态不一致或保护失效
- P2：稳定性、可维护性和体验问题

---

## 一、经济系统

### P0：统一余额权威来源

当前状态：

- SQLite `sfmc_economy_accounts` 是余额唯一权威来源。
- `Money.ts` 仍使用进程内缓存。
- `Money.get()` 只读缓存，未加载时默认返回 `0`。
- UI 大量直接调用 `Money.get()`。
- 缓存没有版本号、加载状态和过期时间。

修复任务：

- [ ] 将缓存结构改为 `{ balance, version, loadedAt, loading }`。
- [ ] 区分 `getCached()` 和 `getFresh()`。
- [ ] 未加载余额不能默认显示为 `0`。
- [ ] 使用数据库返回的 `version` 防止旧请求覆盖新余额。
- [ ] 玩家进入服务器后完成余额加载再允许余额敏感页面使用。
- [ ] 所有交易成功后使用服务端返回的账户快照更新缓存。
- [ ] 土地、合作社商店、红包和转账统一刷新机制。

### P0：补全经济交易响应

- [ ] 土地购买接口返回最终余额、账户版本和交易 ID。
- [ ] 土地删除接口返回最终余额、账户版本和交易 ID。
- [ ] 普通 credit/debit 返回完整账户快照。
- [ ] transfer 返回 source 和 target 双方的最终账户快照。
- [ ] 统一解析经济 API 响应。
- [ ] 不再让业务模块自行猜测交易后的余额。

### P0：增加交易幂等性

问题：

- HTTP 超时不代表数据库事务没有提交。
- 客户端重试可能重复扣款或重复发放。

任务：

- [ ] 所有写交易增加 `idempotencyKey`。
- [ ] 数据库为幂等键建立唯一约束。
- [ ] 重复请求返回原交易结果。
- [ ] 土地购买、删除、转账、红包、合作社商店交易全部使用幂等键。

### P0：限制账户操作权限

问题：

- 服务端信任请求体中的 `actorId`、`sourcePlayerId` 和 `targetPlayerId`。
- 任何获得接口访问权的本地调用者都可能伪造玩家身份。

任务：

- [ ] 明确 db-server 与 SAPI 的信任边界。
- [ ] 普通账户操作限制为操作者本人。
- [ ] 管理员调整余额使用独立接口和明确权限。
- [ ] 禁止普通 credit/debit 任意指定其他玩家账户。
- [ ] 敏感 GET 接口增加认证要求。
- [ ] 统一使用 DB token 或受限服务身份。

### P0：修复领域交易非原子问题

#### 合作社商店

- [ ] 购买必须保证扣款、扣库存、发放物品一致。
- [ ] 发放失败时不能永久扣款。
- [ ] 出售必须保证扣物品和加钱一致。
- [ ] 加钱失败时不能永久丢失物品。
- [ ] 优先将商店交易移动到 db-server 或实现可靠的补偿状态。

#### 红包

- [ ] 创建红包必须同时完成扣款和红包创建。
- [ ] 领取红包必须同时完成红包状态更新和账户入账。
- [ ] 防止并发领取同一份红包。
- [ ] 防止红包状态已更新但入账失败。
- [ ] 增加过期红包退款策略。

#### 合作社银行

- [ ] 存款必须同时完成玩家扣款和合作社余额增加。
- [ ] 取款必须同时完成合作社余额减少和玩家入账。
- [ ] 防止中间请求失败造成账实不一致。
- [ ] 增加合作社银行事务流水。

### P1：移除不安全的 `Money.set()`

问题：

```text
读取余额 -> 计算差额 -> 调用 add
并发交易可能导致最终余额错误。
任务：
- 删除或限制 Money.set()。
- 如必须设置绝对余额，使用服务端 compare-and-set 或管理员专用接口。
- 所有业务使用 credit/debit/transfer 语义。
P1：统一交易流水模型
- 双账户转账分别记录 source 和 target 的余额变化。
- 流水包含 transaction ID、reference type、reference ID。
- 流水包含 source/target before/after balance。
- 交易失败也记录必要的失败原因。
- 所有领域交易可通过 reference ID 追踪。
二、领地系统权限与数据安全
P0：移除通用 managers 写入逻辑
问题：
- 普通土地名称或保护设置更新会提交 managers。
- 服务端可能删除所有普通成员并重新插入为 Admin。
任务：
- 从普通土地更新请求移除 managers。
- 服务端通用 PATCH 禁止修改成员。
- 删除或废弃 managers 兼容字段。
- 成员角色只能通过专用成员 API 修改。
- 增加角色保持回归测试。
P0：修复折扣购买退款套利
问题：
- 购买使用折扣价格。
- 删除使用未折扣基础价格。
- 退款比例和价格配置变化会影响历史土地。
任务：
- 土地保存实际成交价格。
- 土地保存购买时退款比例。
- 删除按历史成交价计算退款。
- 为旧土地设计迁移策略。
- 增加折扣购买、删除退款回归测试。
P0：修复并发删除重复退款
任务：
- 删除事务内重新读取 active 土地。
- 条件更新必须检查实际影响行数。
- 只有成功软删除的请求才能退款。
- 增加并发删除测试。
- 增加删除与转让并发测试。
P0：修复并发转让 Owner 不一致
任务：
- 转让事务内重新读取当前 Owner。
- 转让条件绑定当前 Owner。
- 转让前清理旧 Owner 角色。
- 增加唯一 Owner 数据约束或一致性检查。
- 增加并发转让测试。
P1：统一成员过期权限
问题：
- SAPI 和 db-server 对 expires_at 判断不一致。
- 过期 Admin 可能仍能执行部分土地管理操作。
任务：
- SAPI 角色判断检查 expires_at。
- db-server 管理权限检查检查 expires_at。
- 邀请接受、角色变更、移除成员和撤销邀请统一使用过期规则。
- 增加过期成员测试。
P1：绑定选点申请维度
任务：
- 申请会话保存 dimensionId。
- 禁止跨维度设置第二点。
- 确认购买使用申请会话维度。
- 玩家离开时清理会话。
- 增加会话过期时间。
- 增加跨维度手动验证。

- GUI 显示具体可操作错误。
- 网络错误与业务拒绝分开处理。
